# Public workflow reference

This is the external-share version of a human-approved Threads content operation.

## Structure

```text
research / account performance check
→ candidate drafting
→ approval gate
→ publishing
→ verification
→ response analysis
→ next-post adjustment
→ operating log
```

## Why this works

Most social posting systems stop at drafting. This workflow treats posting as a small operating system:

1. **Input** — current account signals, external post shapes, and business goal.
2. **Decision** — the human approves what gets published.
3. **Action** — only approved content is posted.
4. **Verification** — the live post and all reply segments are checked.
5. **Learning** — comments and metrics shape the next post.

## Recommended post structure

For many practical accounts, a 2–3 segment format works well:

```text
1/3 main insight or problem
2/3 example or explanation
3/3 concrete CTA or question
```

The CTA should ask for one concrete action, not several.

Good CTAs:

```text
Comment with the step that still takes the longest.
DM me the word CHECKLIST if you want the template.
Save this and test it on one task today.
```

Weak CTAs:

```text
What do you think?
Follow for more.
Let me know your thoughts.
```

## Verification principle

Publishing is not done when the button is clicked. It is done only when the live post is checked.

Verify:

- canonical URL
- visible hook
- reply count / segment count
- final CTA
- no duplicated or missing tail segment

## Public safety checklist

Before sharing a workflow externally, remove:

- private account IDs
- local filesystem paths
- browser profile names
- raw CDP endpoints
- API keys, cookies, tokens, and passwords
- private Discord/Telegram routing information
- client/customer data
- unpublished business strategy

Use generic examples and placeholders instead.
